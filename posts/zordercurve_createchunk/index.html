Z階層曲線から任意範囲を効果的に探索する方法のメモ

はじめに#

  • このポストは前提として「Z階層曲線（Morton Code・Z-Order Curve）」を知っていることを要します。
  • 添付コードは全てRustで書いています。
  • あくまでもメモとして書いたので、ざっくりとした感じがするところはご了承ください。
  • 実装の前提が違う場合には次の説明が適用できなくなる可能性があります。

説明#

3D空間でz曲線を使って空間の最適化を行うので、空間の構造体は次のように書きました。

struct Oct<T> {
    datas: Vec<T>,
    side_length: usize,
}

データリストはX…Y…Z…順ではなく、そのままZ-Order Curveの順（0,1,2,3…XYZ）に保存させます。じゃないと任意の空間情報が渡されて、それがいくつかのChunk（塊）に分割されたとしても無駄になってしまいます。

Z-Order Curveのコンテナを生成する時に渡すサイズの値は2のべき乗にする必要があります。そしてZ-Order Curveのコンテナに定数時間で各要素に接近するために必要となるインデックスに、XYZ軸3つの軸からの位置情報が全部入るためにサイズは1 << 21 - 1までに制限されます。

なぜ21かと言いますと、8Bytesのインデックスに3つの軸の値を均一に保存するためには21桁のビットまでが最大になるからです。

impl<T: Default + Clone> Oct<T> {
    pub fn with_length(len: usize) -> Option<Self> {
        const LIMIT: usize = (1 << 21) - 1;
        if len > LIMIT || !len.is_power_of_two() {
            None
        } else {
            // ...make datas: Vec<T> with len * len * len size.
            Some(Self {
                datas,
                side_length: len,
            })
        }
    }
}

まず、範囲でコンテナに接近することを見る前に各単一位置からコンテナの適切な位置に接近することから見せたいと思います。

単一3Dインデックスに当てはまるところに値を更新する関数は次となります。

// 完全なコードではなく一部分だけを抜粋して再構成しました。
pub fn set_value(&mut self, indices: Index3, value: T) {
    let index = encode(indices);
    self.datas[index] = value;
}
// ↓
fn encode(&self, indices: Index3) -> usize {
    let x = convert_order_bits_3d(indices.0);
	let y = convert_order_bits_3d(indices.1);
	let z = convert_order_bits_3d(indices.2);
	(y << 2) | (z << 1) | x
}

入れたインデックス indicesはZ-Order Curveのコンテナに接近するために特殊なインデックス値に変換します。ここではその過程をEnconding（エンコーディング）と名付けました。

encode関数では各軸の位置をコンテナで使えるためのビット値に変換しています。この各軸からのビット値をZ-Order Curveの進行方向次第にビットシフトし一つに合わせると、結果にコンテナに接近が可能になるインデックス値が返されます。

convert_order_bits_3d関数は次のような構成となります。

fn convert_order_bits_3d(index: usize) -> usize {
    let index = index & ((1 << 21) - 1);
    (0..21)
        .rev()
        .fold(0, |r, v| (r << 2 | (index & (1 << v)))
}

この関数ではインデックス値をコンテナのリストに接近するための各軸のビット値に変換します。

例えば、1を入れたら返し値は0b001になります。もし15を入れたら0b001001001001になります。つまり、3つの軸からの数値をビットたちが集まって揃うように間を置きます。逆変換する時にはこの特徴を逆として活かせば簡単に実装できます。

さて範囲を決めて値を更新する関数は次となります。

// このコードではエラーチェックは省略します。
pub fn set_values(&mut self, start: Index3, len: Index3, value: T) -> Result<()> {
    let inclusive_len = {
        let end = Index3::from_scalar(self.len);
        cmp::min(start + len - Index3::from_scalar(1), end) - start
    };
    let inclusive_end = start + inclusive_len;
    
    // Update.
    let chunks = self.to_contiguous_chunks(start, inclusive_end);
    chunks.into_iter().for_each(|(start, len)| {
        self.datas.iter_mut()
        	.skip(start).take(len)
        	.for_each(|v| *v = value.clone());
    });
    Ok(())
}

上の関数ではスタートと到達点のインデックスではなく、スタートと最後までの距離を取りました。そのほうがエラーチェックがしやすいし、スタートから与えた分だけ特定方向へ範囲を展開しますというのがはっきりしているからです。

与えられた任意の範囲からZ-Order Curveのリストに連続な範囲として切り取るには、to_contiguous_chunksメソッドを呼びます。Startの位置と、自分を含むゴールの位置を入れます。（閉区間になります）

fn to_contiguous_chunks(&self, start: Index3, end: Index3) -> Vec<(usize, usize)> {
    let mut results: Vec<(usize, usize)> = vec![];
    let mut pendings = vec![(self.encode(start), self.encode(end))];
    while !pendings.is_empty() {
        let (start, end) = pendings.pop().unwrap();

        let is_same = start == end;
        let is_contiguous = 
                (start ^ end).overflowing_add(1).0.is_power_of_two()
            && 	(end - start).overflowing_add(1).0.is_power_of_two();
        if is_same || is_contiguous {
            results.push((start, end - start + 1));
            continue;
        }

        let split = Self::get_split_mode(start, end);
        let (start, end) = (self.decode(start), self.decode(end));
        let (little_max, big_min) = match split {
            ESplit::Horizontal => {
                let bm = get_bigmin(start.z, end.z);
                (Index3(end.x, end.y, bm - 1), Index3(start.x, start.y, bm))
            }
            ESplit::Vertical => // zに対しHorizontalコードを繰り返す
            ESplit::Depthical => // yに対しHorizontalコードを繰り返す
        };

        pendings.push((self.encode(start), self.encode(little_max)));
        pendings.push((self.encode(big_min), self.encode(end)));
    }

    results
}

ここで入れられた元の範囲を分解しつつ、連続になった範囲ができたらそれを resultsに入れて返します。取り出した範囲を再度区切らない条件は以下となります。

  • そもそも範囲の最初と最後が同じ位置であること
  • z-orderで範囲がそのまま連続していて、かつ長さが2のべき乗になること

その他は範囲をビット次第に特定の軸で2頭分して二つの範囲の長さを2のべき乗になるように近づけます。ここでLitMaxとBigMinという分割過程で生み出される新しい位置があり、分かれた2つの範囲は[Start, LitMax]と[BigMin, End]の範囲を持ちます。

肝心なところは、このLitMaxとBigMinをどうやってちゃんと生成するかですね。

私が読んだ論文によると、与えた領域から一つの軸を取り、まずそれをちゃんとした塊に近づけられるインデックスを求めます。そうしたらLitMaxとBigMinの他の二つの軸の値は逆にすればいいだけになります。

Startからは2分化された領域の一つ目の終わり位置をLitMaxにされ、逆にBigMinはその次から始まる2つ目の領域の最初位置になって二つに分けることができるそうです。

まず軸を決めることから始めます。上のコードではget_split_mode(start, end)で軸を決めます。

fn get_split_mode(start: usize, end: usize) -> ESplit {
    let v = start ^ end;
    let msb = msb_of(v);
    match msb % 3 {
        2 => ESplit::Depthical,
        1 => ESplit::Horizontal,
        0 => ESplit::Vertical,
        _ => panic!(),
    }
}

上の通常のインデックスからリストに接近するために特殊ないインデックスに変換するロジックがありましたが、そこでは0ビット順からXZYXZYXZY…順に積み重ねるようにしています。XZY順をそのままこのロジックでも適用しちゃいます。

どの軸を切るかが決まったら、matchでLitMaxとBigMinを算出します。

let (little_max, big_min) = match split {
    ESplit::Horizontal => {
    let bm = get_bigmin(start.z, end.z);
        (Index3(end.x, end.y, bm - 1), Index3(start.x, start.y, bm))
    }
    ESplit::Vertical => // zに対しHorizontalコードを繰り返す
    ESplit::Depthical => // yに対しHorizontalコードを繰り返す
};

実装したコードではget_bigmin()関数でBigMinのインデックス位置を決めます。他の二つの軸は、両端の軸の値を取り替えることで自動で二分化されます。

fn get_bigmin(start: usize, end: usize) -> usize {
    let mut v = 0;
    loop {
        let msb = msb_of((start | end) & !v);
        v |= 1usize << msb;

        if start < v { return v; }
    }
}

get_bigmin()関数ではstartインデックスとendインデックスの真ん中に存在する、2のべき乗に近いインデックスを返します。論文ではこの項目について「all common most significant bits of values」だけ書いてあったので、意味がちゃんと伝わらず手探りで考えて書けました。

最初は[Start, LitMax]と[BigMin, End]を待機リストに入れて、連続だとしたら結果リストに入れることを繰り返し、待機リストが空になると終わりとなります。

おわりに#

ちなみにアルゴリズムを使わずに、単純に最初から最後までインデックスを一つずつ辿りながら値を更新しようとした時との性能差は次の通りでした。（インデックスをそのまま任意方向に移して値を更新する方法もあったりします。）

fn setup() -> Oct<u32> {
    let len = 64;
    let mut mat = Oct::<u32>::with_length(len).unwrap();

    for y in 0..len {
        let yv = (y * len * len) as u32;
        for z in 0..len {
            let zv = (z * len) as u32;
            for x in 0..len {
                let value = x as u32 + zv + yv;
                mat.set_value(Index3(x, y, z), value).unwrap();
            }
        }
    }
    mat
}

#[bench]
fn use_oct_naive_optimal(b: &mut Bencher) {
    let mut mat = setup();
    b.iter(|| {
        mat.set_values_naive(Index3(8, 8, 8), Index3(48, 48, 48), 0).unwrap();
    });
}

#[bench]
fn use_oct_split_optimal(b: &mut Bencher) {
    let mut mat = setup();
    b.iter(|| {
        mat.set_values(Index3(8, 8, 8), Index3(48, 48, 48), 0).unwrap();
    });
}

#[bench]
fn use_oct_naive_heavy(b: &mut Bencher) {
    let mut mat = setup();
    b.iter(|| {
        mat.set_values_naive(Index3(11, 9, 7), Index3(31, 43, 37), 0).unwrap();
    });
}

#[bench]
fn use_oct_split_heavy(b: &mut Bencher) {
    let mut mat = setup();
    b.iter(|| { 
        mat.set_values(Index3(11, 9, 7), Index3(31, 43, 37), 0).unwrap(); 
    });
}

Cargo benchから実行したベンチマーク結果は以下となりました。

running 4 tests
test use_oct_naive_heavy   ... bench:   6,102,430 ns/iter (+/- 30,576)
test use_oct_naive_optimal ... bench:   6,349,580 ns/iter (+/- 61,319)
test use_oct_split_heavy   ... bench:     655,900 ns/iter (+/- 8,541)

上のテストコードでは各軸64個、つまり262144個のリストに対してテストをやっています。コンテナのリストの要素の数が小さければ単純にする方法と差はあんまり出てこないかもしれません。

が、上の結果ではHeavy（連続する範囲がほぼない状態）でも通常より10倍以上の速度になっていたことがわかりました。もちろん最初からほぼ連続になれる状態の範囲の場合にはとてつもなく速くなります。
